命令

create table

create table instructor
(ID varchar (5)
name varchar (20) not null
dept_name varchar (20)
salary numeric (8,2)
primary key (ID)
foreign key (dept_name) references department)

  • 分号可选
  • 主键非空且唯一

E-R模型部分

实体-联系模型
  • 实体：一个可区分的对象，比如人可通过身份证号区分
    • 通过属性来表示 每个属性有一个值
  • 实体集：一类拥有相同属性的实体集合
  • 联系：多个实体之间的相互关联，比如A和B是师生关系
  • 联系集：相同类型联系的集合，比如师生关系集
  • 参与：实体集之间的关联，实体集E参与联系集R
  • 角色：一个实体的代称
  • 描述性属性：一个联系本身具有的属性
    • 如师生关系联系中的“开始日期”
  • 多值属性：数据类型为数组的属性“{phone_number}”
  • 二元联系：涉及两个实体集的联系集
  • 多元关系：涉及多个实体集的联系集
    • 度：参与的实体集数目
属性
  • 简单属性：不可能划分为更小部分
  • 复合属性：可以再划分为更小的部分
  • 派生属性：可以由现有属性求得的隐藏属性
    • 比如当存有出生日期时，通过现在的日期就可以求得年龄，此时年龄就是派生属性
    • 可以在属性里记为age（）代表函数计算可得
映射基数
  • 一对一：一个a与一个b关联 一个b与一个a关联
  • 一对多：一个a与任意b关联 一个b与一个a关联
  • 多对多：一个a与任意b关联 一个b与任意a关联
参与约束
  • 全部参与：实体集中每个实体都参与到联系集中至少一个联系
  • 部分参与：有实体没参与联系集中的联系
键
  • 实体集的主键就是关系的主键
  • 联系集的超键是与之关联的所有实体集的主键还有自身关联属性主键的并集（但不一定是主键）
    • 如果不是多对多的映射，则只需要“多”的那一边实体集的主键即可（可以做到唯一确定）
      • 如多个学生对一个老师 则确定一个学生即可确定一段师生关系
    • 如果是多对多的映射，则需要所有主键并集共同确定
删除冗余属性
  • 如果两个实体集中有重复的属性 删除非主键实体集中的该属性
    • 如果两个实体的重复属性已经用联系关联在一起了就可以把其中不重要的删掉
E-R图

  • 元素

    • 双矩形：实体集名字-实体集属性
    • 菱形：联系集
    • 单矩形：联系集的相关属性
    • 线段：关联实体集和联系集
    • 虚线：关联联系集和其属性
    • 双线：实体的参与度
    • 双菱形：若实体集的标志性联系集

  • 映射

    • 一对一：A↔️B
    • 一对多：A⬅️B（A可以有多个B B只能有一个A）
    • 多对多：A—B
    • 把箭头理解成“追求” 互相追求就是一对一 被追求的可以有多个 想追求的只能有一个
    • 基数约束0..* :我面对的人有多少个
      • 理解成“相亲的名片”

  • 角色

    • 在线段上取一个别名 一般用于自己和自己联系

  • 弱实体集

    • 当两个属性都保存了一个属性 并且该属性是两个属性的主键成员的时候 也要删掉其中一个 被删的实体集会变成“弱实体集”因为它的主键已经不能唯一标识每一个实体（不配做主键）
    • 因此缺的这个实体集会“依赖”没缺的实体集，被称作标识实体集，称标识实体集“拥有”弱实体集，他们之间的联系称为标识性联系
    • 而弱实体集“残破”的主码被称为分辨码或部分码
      • 用虚线标注
    • 因此弱实体集的主码是分辨码和标识实体集的主码组合而成（缺的补回来）
    • 产生的原因本质是因为E-R关系联系至上，不允许任何已经被抽象成联系的冗余信息存在
      E-R图示例
E-R转换为关系模式
  • 每一个实体集一个关系模式 每一个联系集一个关系模式
  • 简单属性-强实体集
    • 实体集名变模式名
    • 简单描述属性变成模式的属性
    • 实体集主键变为模式主键
  • 复杂属性-强实体集
    • 复合属性：把复合属性拆开，变成多个简单属性存进模式
    • 多值属性：把多值属性从原属性移除，单独创建一个关系，命名为“原实体集名_多值属性名” 属性为原实体集主键+多值属性 且均为主键，这个关系的每一个元组对应原属性的每一个值，最后再把原实体集主键的参照关系添加了
      • 例如原先instructor的phone_num为多值
      • instructor中去除phone_num 新建关系instructor_phone(ID, phone_number)且两个属性均为主键
      • ID是参照instructor表的外键
      • 如果原实体集只有俩属性 把多值属性移除了就剩一个主键的话 原实体集就不用单独转化了 直接保留这个单独生成的关系就行
  • 弱实体集
    • 主码就是自己的部分码拼上属主实体集的主码
    • 然后把自己向属主借来的主码建立外码约束和级连删除
  • 联系集
    • 主键就是联系集的主键
      • 自己的属性+如果有“多”就拿多的主键
        • 多对多两个并 一对一俩都行
      • 综合一元和多元
        • 谁没箭头选谁 都没箭头都得选
    • 所有非自身的属性都建立外码约束
    • 自身与自身的联系把两个角色名都当作属性名且都当作主键 且都要参照回同一个实体集
    • 二元关系都有两个外码约束 代表着联系集两端的实体集
  • 模式冗余：强弱实体集之间的联系集一般都是冗余的，其信息已经被扩充过的弱实体集完全替代
  • 模式合并：
    • 如果A对B是多对一 且A完全参与 则A和AB可以合并，属性取并集
      • 比如instructor对department是多对一并且instructor完全参与 则instructor可以和inst_dept合并
      • instructor新增dept_name属性
    • 一对一则都可以合并
    • 联系集的主键在谁哪里就跟谁合并
    • 不完全参与的合并了会有null
设计问题
  • 实体集还是属性？
    • 多值属性如果还有其他关联或者作用设为实体
    • 只是属性作用作为属性
  • 实体集还是联系集？
    • 描述实体之间的行为的时候采用联系集
  • 二元集还是n元集？
    • 所有的n元集联系都可以修改为二元集联系
    • 但n元集的联系更加清晰
    • n元集约束不一定能修改为二元集约束
  • 联系属性
    • 联系属性可以并入一对多的“多”实体集
    • 比如师生关系的日期可以作为学生的属性，因为学生可以唯一确定关系
扩展特性

  • 特化

    • 其实就是子类/子集/划分
    • 重叠特化
      • 可以以多种不同标准划分 一个实体可以有多种特化身份
      • 分开独立的两个箭头
    • 不相交特化
      • 一般不能用多种标准 一个实体只能最多有一种特化身份
      • 聚集成一簇的一个箭头

  • 概化

    • 特化的逆过程，就是子集找共同点变成父集
    • 条件定义
      • 进入哪个特化由属性决定 比如研究生和本科生
    • 用户定义
      • 进入哪个特化由人为决定 比如人为分组

  • 继承

    • 高层实体属性被底层继承
    • 可以单继承可以多继承

  • 全部概化/特化

    • 所有高层都属于某个底层实体
    • 所有学生一定是研究生或者本科生

  • 部分概化/特化

    • 有的高层没有对应的底层实体

  • 聚集

    • 把一个联系集和实体集看成一整个高层实体与别的实体做联系
拓展特性转化为关系模式
  • 概化
    • 低层实体有自己的所有属性+高层实体的主键
    • 底层实体的主键设为高层实体的主键
    • 建立外码约束
    • 如果是完全划分 则不需要高层实体 每个底层实体有自己的所有属性+高层所有属性，主键还是高层主键
      • 这种会有外码约束问题
  • 聚集
    • 聚集的主键是中心联系集的主键，用联系集对应的关系就可以
